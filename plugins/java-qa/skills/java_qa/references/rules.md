# QA Automation — Rules (quality gates)

Три свода правил для долгоживущей Java QA-автоматизации. Применять к новым/мигрированным тестам, рефакторингу фреймворка, изменениям CI-линий, CLI/MCP-инструментам и AI-сгенерированной автоматизации.

---

## Rule 1 — Test Code Quality

Держать Java-тесты читаемыми, диагностичными и устойчивыми к безопасному рефакторингу прода.

1. Тест обязан называть поведение, условие и ожидаемый результат.
2. Ассерты доказывают бизнес-смысл, а не только существование объекта.
3. Тестовые данные читаемы и локальны, если общие фикстуры не выбраны намеренно.
4. Моки служат изоляции, а не маскировке неясного дизайна.
5. Имена тестов помогают быстро локализовать упавшее поведение.
6. Setup не больше, чем проверяемое поведение, без явной причины.
7. Ожидаемый результат прослеживается до требования, контракта, бага или доменного правила.

**Review-вопросы:** что защищает тест? что сломается в проде, если он упадёт? ассерт достаточно точен для диагноза? тест не слишком близок к деталям реализации? можно ли сделать меньше/быстрее?

**Плохие признаки:** `assertNotNull` как главный ассерт; большие копипаст-фикстуры с неясными важными полями; много моков и нет видимого поведения; имя теста об одном, ассерты о другом; скрытая зависимость от порядка/окружения.

## Rule 2 — Flaky Test Control

Сделать флаки наблюдаемыми, диагностируемыми и чинимыми, а не нормализовать нестабильный CI.

1. Классифицировать флак до добавления ретраев.
2. Записывать failure mode, окружение, seed, время, браузер, драйвер, версии зависимостей.
3. Заменять фиксированные ожидания на condition-based waits.
4. Убирать общее мутабельное состояние из параллельных тестов.
5. Отделять медленные сбои окружения от продуктовых сбоев.
6. Карантинить только с владельцем и датой возврата.
7. Предпочитать детерминированный setup и явный cleanup.

**Частые источники флака:** допущения о времени/таймзоне; случайные данные без захвата seed; async со слабой стратегией ожидания; утечка состояния БД/очереди; внешние HTTP-зависимости; рассинхрон браузера и драйвера; зависимость от порядка.

## Rule 3 — Automation Decision Quality Gates

Не пускать малоценную, хрупкую или вводящую в заблуждение автоматизацию в долгосрочный набор/агентный тулчейн. Семь ворот:

1. **Behavior gate** — автоматизация называет защищаемое поведение. Reject/defer, если ожидаемое поведение неизвестно, единственный оракул — флаки/неясный тест, тест проверяет только реализацию или только факт исполнения кода.
2. **Layer gate** — выбранный слой обоснован; предпочесть дешёвый слой при равной уверенности.
3. **Oracle gate** — каждый ассерт прослеживается до требования, API-контракта, OpenAPI, consumer/provider-контракта, регрессионного правила бага, доменного инварианта или явного решения стейкхолдера. Tribal/выведенное — помечать как assumption.
4. **Determinism gate** — определены setup данных, изоляция, cleanup, контроль зависимостей. Блок при общем мутабельном состоянии, зависимости от порядка, неконтролируемой third-party, произвольных sleep, грязных персистентных данных, недокументированных ручных предусловиях.
5. **Diagnostics gate** — вывод при падении помогает классифицировать причину (входы, expected vs actual, категория сбоя, контекст окружения).
6. **CI placement gate** — у каждого теста/инструмента своя линия: PR fast lane / smoke / nightly regression / release gate / manual-exploratory / quarantine (owner+expiry).
7. **Agent-safety gate — ТОЛЬКО если результат = CLI/MCP/agent-facing инструмент** (не для обычного JUnit/API-теста!): стабильный структурный вывод, типизированные категории сбоя, редакция секретов, политика мутации/cleanup, путь переиспользования через общее ядро. Для обычных тестов этот gate неприменим — к ним применяется Diagnostics gate (№5).

**Метки решения** (canonical enum, общий со `decision-flow.md` и `workflows.md`):
`DO` · `DO (WITH RISK)` · `SPIKE FIRST` · `LATER` · `DO NOT DO` · `MANUAL FOR NOW`.

**Блокировать:** автоматизацию без названного поведения; status-only API-проверки без бизнес/контракт-результата; UI/E2E для логики уровня unit/component/API/contract; общее мутабельное состояние без изоляции; prompt-only там, где нужны стабильные инструменты; карантин без владельца/срока.

## Rule 4 — Schema Evolution / Compatibility (tolerant reader vs fail-fast)

Не маскировать рассинхрон контракта и не делать систему хрупкой. Граница, а не лозунг.

1. **Не подавать `@JsonIgnoreProperties(ignoreUnknown=true)` / отключение `FAIL_ON_UNKNOWN_PROPERTIES` как «фикс падений».** Это политика эволюции схемы, а не защита.
2. **Tolerant — только к неизвестным полям, которые КОНКРЕТНЫЙ десериализатор по контракту не использует:** провайдер — к лишнему в request; консьюмер — к новым полям в response. Одно правило для обеих сторон (не «только в ответе»).
3. **Fail-fast** — на семантически обязательном: required-поле, неизвестный enum в используемом поле, смена типа, удалённое/переименованное используемое поле.
4. **Fail-fast надо РЕАЛИЗОВАТЬ — он не возникает сам.** Для required: `@JsonProperty(required=true)` + `FAIL_ON_MISSING_CREATOR_PROPERTIES` работают **только для creator-свойств** (конструктор/`@JsonCreator`/record), НЕ для setter/field DTO; для mutable — Bean Validation / schema / contract-тест. Отказ от primitive. `required=true` сам по себе не универсальная защита. Место — CI (provider verify / schema-diff), не рантайм прода.
5. **Защита от рассинхрона = тест, не аннотация.** `ignoreUnknown` не заменяет контракт-тест.
6. **Jackson-нюансы (без иллюзий):**
   - `ignoreUnknown` **НЕ рекурсивен** — не покрывает вложенные DTO/элементы списков. Политика на каждом nested, mix-in, либо настройка `ObjectMapper` консьюмера (предпочтительно — одно место).
   - **маскирует rename/delete** используемого поля через тихий дефолт (`null`/`0`/`false`). Поэтому required-поля **не делать primitive** — потеря неотличима от нуля.
   - **НЕ маскирует** смену типа и unknown enum в используемом поле — там падает само.
7. **Rollout-порядок:** сначала tolerant-reader на ВСЕ консьюмеры, ПОТОМ провайдер шлёт новые поля. Аннотация в новой версии не помогает уже задеплоенному старому консьюмеру.
8. **Unknown enum — не всегда fail-fast:** бизнес-ветка → fail-fast; extension point с безопасным `UNKNOWN` → tolerant легитимна. Решать по семантике.

**Review-вопрос-маркер:** если предлагаешь заглушить unknown-поля — спроси «это аддитивное поле, которое этот десериализатор не читает (tolerant ОК), или это маскировка breaking change, который должен ловить тест (fail-fast)?». И проверь: не primitive ли required-поле, покрыты ли nested, какой порядок выкатки. Без ответов — не навешивать.
