# QA Automation — Workflows

Два структурированных ревью-процесса. Запускать ДО внедрения изменения в долгосрочный набор.

---

## Workflow 1 — Automation Review

Ревью изменения Java QA-автоматизации до попадания в регрессионный набор.

**Вход:** диф/дизайн теста · источник продуктового поведения · связанные существующие тесты · CI-задача, где побежит тест · известная история флака.

**Шаги:**
1. Определить защищаемое поведение.
2. Определить выбранный слой теста.
3. Проверить, даёт ли более дешёвый слой ту же уверенность.
4. Проверить setup и cleanup данных.
5. Проверить силу ассертов и диагностику падения.
6. Проверить допущения о зависимостях и окружении.
7. Оценить влияние на runtime и стабильность CI.
8. Зафиксировать набор: smoke / PR / nightly / release.

**Output:**
```markdown
# Automation Review
## Behavior protected
## Current layer
## Suggested layer
## Assertion quality
## Test data and environment
## Flakiness risks
## CI placement
## Required changes
```

**Критерии одобрения:** ясный источник поведения; обоснованный слой; полезное сообщение о падении; детерминированный для CI setup; нет дублирования другого теста без причины.

## Workflow 2 — QA Automation Architecture Decision

Структурированное ревью ДО архитектурного изменения QA-автоматизации. Применять для: выбора фреймворка, миграции набора, выбора слоя тестов, дизайна общей тест-платформы, изменений CI-линий, service virtualization, contract testing, экспозиции CLI/MCP, AI-agent тулинга.

**Вход:** предложение решения · защищаемое поведение/возможность платформы · источник истины · текущая реализация и боль · существующие тесты и CI-линии · история флака/инцидентов · ограничения тулинга и окружения · нужды agent/CLI/MCP-консьюмера.

**Шаги:**
1. Сформулировать решение одним предложением.
2. Определить защищаемое поведение/возможность.
3. Определить источник истины и классифицировать допущения.
4. Картировать текущее покрытие и боль.
5. Выбрать самый дешёвый достаточный слой теста/тулинга.
6. Проверить архитектурную пригодность и путь переиспользования.
7. Проверить данные, состояние, cleanup, детерминизм.
8. Проверить силу ассертов и диагностику.
9. Проверить CI-placement и влияние на runtime/стабильность.
10. Проверить agent-readiness, если тулинг отдаётся AI-агентам.
11. Сравнить альтернативы (≥2 при архитектурном решении).
12. Решить **canonical-меткой** (`DO` / `DO (WITH RISK)` / `SPIKE FIRST` / `LATER` / `DO NOT DO` / `MANUAL FOR NOW` — общий enum со `decision-flow.md`/`rules.md`) и определить proof.

**Output:**
```markdown
# QA Automation Architecture Decision
## Decision        DO / DO (WITH RISK) / SPIKE FIRST / LATER / DO NOT DO / MANUAL FOR NOW
## Proposal
## Protected behavior or capability
## Source of truth
## Recommended layer or architecture
## Alternatives considered
## Risks and mitigations
## Implementation slice
## Proof
## Follow-up artifacts
```

**Критерии одобрения:** названо защищаемое поведение; слой обоснован против дешёвых альтернатив; источник истины явный; состояние и cleanup детерминированы для CI; диагностичный вывод; явный CI-placement; агентные инструменты дают стабильный структурный вывод и не дублируют бизнес-логику.
